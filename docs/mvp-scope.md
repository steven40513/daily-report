# MVP 功能範圍 — 營建日報系統 v1

**日期：** 2026-06-27
**目標時程：** 1-2 週
**首批用戶：** 1-2 位工地主任試用
**技術方案：** 單一 HTML + JavaScript，瀏覽器端完成所有功能

---

## 產品定位

為了 **50-60 歲的工地主任**，
他們在 **收工後 10-15 分鐘** 需要填寫每日施工報告，
本系統是一個 **手機端日報填報工具**，
能夠 **快速填完日報並產出 PDF 透過 LINE 分享**，
不同於 **紙本手寫或 Excel 填寫**，
我們的優勢在於 **帶入昨天資料，只改有變動的部分，3-5 分鐘填完**。

---

## MVP 做什麼

### 功能 1：專案設定（一次性）

**用戶故事：** 身為工地主任，我想要設定專案基本資訊，以便每天的日報自動帶入這些固定資料。

**範圍：**
- 專案名稱、公司名稱
- 開工日期、竣工日期
- 簽核人員姓名（董事長、總經理、工務協理、工地主任等）
- 工種分類預設（裝修/結構/基礎/機電/假設工程）
- 材料預設清單（6 大類，含預設品項）
- 機具預設清單（4 大類，含預設品項）

**驗收標準：**
- [ ] 設定的資料存入 localStorage，重啟瀏覽器不遺失
- [ ] 設定頁的工種分類可新增、刪除
- [ ] 設定頁的材料/機具品項可新增（在對應分類下）
- [ ] 設定的專案名稱、公司名稱會反映在 PDF 輸出中

**邊界情況：**
- 首次使用時載入預設值（原型中的預設資料）
- 設定頁儲存前不影響已填寫的日報

---

### 功能 2：五步日報填寫

**用戶故事：** 身為工地主任，我想要用簡單的步驟填寫每日施工報告，以便在 10 分鐘內完成。

**步驟 1 — 天氣與施工天數**
- 日報日期（預設今天，可修改）
- 上午/下午天氣選擇（晴/陰/雨，互斥）
- 日曆天數（從開工日期自動計算）
- 施工天數（可 +/- 調整）

**步驟 2 — 今日人員**
- 各工種人數（+/- 按鈕調整）
- 底部即時加總顯示「今日總人數」
- 支援新增工種

**步驟 3 — 營建材料**
- 6 大分類切換（混凝土/砌築/填縫黏著/骨材/鋼筋/其他）
- 每類下的品項勾選 + 數量輸入
- 支援在各分類下新增品項
- 切換分類時保留所有勾選與數量

**步驟 4 — 機具設備**
- 4 大分類切換（挖掘/破碎/吊掛/運輸）
- 操作同材料

**步驟 5 — 進度與備註**
- 今日進度（textarea + SOP 提示）
- 明日進度（textarea + SOP 提示）
- 備註（textarea）

**驗收標準：**
- [ ] 5 步可前進/後退自由切換
- [ ] 各步驟的填寫狀態在切換時保留
- [ ] 人員 +/- 操作後，底部總人數即時更新
- [ ] 材料/機具分類切換正確顯示對應品項清單
- [ ] 新增品項使用安全的 DOM 操作（無 XSS 風險）

### 功能 3：localStorage 資料持久化

**用戶故事：** 身為工地主任，我想要填到一半或填完的日報不會因為關閉瀏覽器而消失。

**資料模型：**
```
localStorage key: "daily-report-{YYYY-MM-DD}"
value: {
  date: "2026-06-26",
  weather: { am: "晴", pm: "陰" },
  constructionDays: 25,
  crew: { "裝修工程": 12, "結構工程": 8, ... },
  materials: {
    "concrete": [
      { name: "混凝土 5000psi", unit: "米", checked: true, qty: 24 },
      ...
    ],
    ...
  },
  equipment: { ... },  // 同 materials 結構
  progress: { today: "...", tomorrow: "...", notes: "..." },
  savedAt: "2026-06-26T18:30:00"
}
```

**專案設定資料模型：**
```
localStorage key: "project-settings"
value: {
  projectName: "拾光薈二期新建工程",
  companyName: "群盛營造股份有限公司",
  startDate: "2026-06-01",
  endDate: "2028-06-01",
  signers: { "董事長": "", "總經理": "", ... },
  crewTypes: ["裝修工程", "結構工程", ...],
  materials: { ... },  // 各分類預設品項
  equipment: { ... }
}
```

**驗收標準：**
- [ ] 每個步驟離開時自動儲存當前狀態
- [ ] 重新開啟頁面時，自動載入今天的日報資料（若有）
- [ ] 專案設定的修改即時存入 localStorage
- [ ] localStorage 總容量不超過 3MB（預留空間給未來照片功能）

**邊界情況：**
- localStorage 不可用時（隱私模式），顯示警告提示
- 跨日（23:59 填、00:01 存）按填寫時的日期算

### 功能 4：帶入昨天資料

**用戶故事：** 身為工地主任，我想要今天的日報自動帶入昨天的填寫內容，以便我只需修改有變動的部分。

**帶入邏輯：**
1. 開始新一天的日報時，檢查 localStorage 中是否有昨天的日報資料
2. 若有，自動帶入以下欄位：
   - 天氣：**不帶入**（每天不同）
   - 施工天數：帶入並 **自動 +1**
   - 人員數量：全部帶入
   - 材料勾選與數量：帶入勾選狀態，**數量清空**（每天數量不同）
   - 機具勾選與數量：同材料
   - 今日進度：帶入昨天的「明日進度」作為今天的「今日進度」
   - 明日進度：**清空**
3. 畫面上顯示綠色提示：「💡 已帶入昨日資料，僅需修改有變動的部分」
4. 若無昨天資料，從專案設定的預設值開始

**驗收標準：**
- [ ] 有昨日資料時，自動帶入且顯示綠色提示
- [ ] 施工天數正確 +1
- [ ] 材料/機具的勾選狀態帶入，但數量清空
- [ ] 昨天的「明日進度」自動成為今天的「今日進度」
- [ ] 無昨日資料時，使用專案設定的預設值
- [ ] 帶入後可自由修改所有欄位

**邊界情況：**
- 跳過一天沒填（週末），找最近一筆日報帶入
- 已有今天部分填寫的資料時，不覆蓋（以今天的為準）

---

### 功能 5：PDF 產生

**用戶故事：** 身為工地主任，我想要把填完的日報產成 PDF，以便透過 LINE 分享給主管和同事。

**範圍（第一版只做第一頁）：**
- 標題：營建工程管理日報表
- 專案資訊表格（名稱、公司、天氣、日期、天數）
- 今日/明日進度、備註
- 人員統計（分類列表或圓餅圖）
- 材料清單（只列已勾選且有數量的品項）
- 機具清單（同材料）
- 簽核欄（從設定帶入姓名）

**技術方案：**
- jsPDF 瀏覽器端生成
- 中文字型嵌入（需處理，這是技術風險最高的部分）
- 產生後提供：下載按鈕 + 分享按鈕（呼叫 Web Share API，支援 LINE 分享）

**驗收標準：**
- [ ] PDF 內容完整反映所有填寫的資料
- [ ] 中文正常顯示（不是亂碼或方塊）
- [ ] PDF 在手機上可透過 LINE 分享
- [ ] PDF 列印出來格式正確（A4 尺寸）

**⚠️ 待確認：** PDF 格式是否有公司或法規規定的固定樣式。若有，需調整版面。

**邊界情況：**
- 材料/機具品項超多時，版面要能自動換頁或縮小字體
- 沒有填寫的區塊顯示「（無）」而非空白

## MVP 不做什麼

| 功能 | 延後原因 |
|------|---------|
| 施工紀錄（CRUD + 工時） | 管理層需求，非現場人員核心需求，增加大量 UI 與邏輯複雜度 |
| PDF 第二頁（施工明細） | 依賴施工紀錄功能 |
| 累計工時自動計算 | 技術複雜度高，需要完整的歷史資料結構 |
| 照片上傳 | 需求未定義 + localStorage 容量限制 |
| 歷史日報瀏覽頁面 | 初期可直接在手機檔案管理器看下載的 PDF |
| 精簡版/完整版切換 | 先只做精簡版 |
| 多專案切換 | 初期只服務一個專案 |
| 資料匯出/備份 | 可延後到驗證完需求再做 |
| 代工廠商管理 | 隨施工紀錄延後 |

## 建議開發順序

```
Week 1:
  Day 1-2: 功能 3 — localStorage 資料模型 + 存取邏輯
  Day 2-3: 功能 1 — 專案設定頁（可編輯 + 存入 localStorage）
  Day 3-4: 功能 2 — 五步日報填寫（接上 localStorage 讀寫）
  Day 4-5: 功能 4 — 帶入昨天資料

Week 2:
  Day 1-2: 功能 5 — PDF 產生（⚠️ 需先確認格式）
  Day 3:   整合測試 + 修復
  Day 4:   部署為 PWA（加到手機桌面）
  Day 5:   交給工地主任試用 + 即時修復
```

**技術風險最高的是：** jsPDF 中文字型嵌入。建議 Week 1 就先做一個最小 PoC（只產生一行中文的 PDF），確認技術可行，不要等到 Week 2 才碰。
